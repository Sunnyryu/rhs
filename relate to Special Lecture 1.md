## 특강!!

#### 프로젝트 관련

```
프로젝트
유니크한 제품, 서비스, 결과 등을 만들기 위하여 한시적으로 수행되는 작업
(한정된 기간과 비용 내에서 정해진 자원을 활용하여 완수하고자 하는 과제)

RFP(Request For Proposal): 제안요청서
RFI(Request For Information): 자료의뢰서

사업참여 의사결정

사업 수행 가능 여부, 매출 규모와 이익률, 고객과의 전략적 관계 등을 고려하여 결정 ~~

IT 프로젝트

크게 컨설팅, 시스템 구축(System Integration - SI), 운영 및 유지보수로 구분되어짐


SI(신규개발) VS SM(운영/유지보수)
강 VS 중약 (업무강도)
보통 SI가 기술스펙이 더 빨리 올라감 (기술 SPEC)

규모

소형 : 10명 이내 수행 / 수행기간 6개월 미만

중형 : 100명 이내 수행 / 수행기간 12개월 이내

대형 : 1000명 이내 수행인원, 수행기간 2년 이상

프로젝트의 규모가 작을수록 성공확률은 높아지며, 프로젝트 규모가 커질수록 성공확률이 현저하게 낮아짐!
프로젝트는 관리가 필요함 !

특징

진행사항이 가시적으로 눈에 보이지 않고 개발자 의존도가 높은 속성이 존재!
초기에 사용자 요구사항을 명확히 정의하기 어려움

분석  ->   설계   ->   개발  ->  테스트   ->  안정화
(초기요구 사항)               

프로젝트 실패 요인

1 요구사항의 변경
2 부정확한 기간과 비용 예측  
3 고객이나 사용자 등의 참여 부족
```

```
동 - 서양 프로젝트 차이

서양은 객체 중심으로 프로젝트를 인식하여 계약을 체결하고 수행 중 발생하는 계약 범위 밖의 요구사항은 쉽게 거절할 수 있어 수행 관리가 비교적 용이
동양은초기에 설정한 계약상의 프로젝트 목표도 중요하지만 고객과의 관계 유지가 더 중요 / 고객의 요구에 따라 목표와 범위가 수시로 변동되어 수행 관리가 어려움

프로젝트 관리 영역

목표 / 수단

목표 (Core Area)
범위관리
일정관리
원가관리
품질관리

수단(Facilites Area)
자원관리
으시사소통관리
위험관리
조달관리
이해관계자관리
```

```

```
```

```

잦은 변경이 발생되는 단점이 있음

sprint로 쪼개서 개발함..
```

방식, 근무시간, 회의, 기타 등을 사전에 정해야 함
```
```
유스케이스 명세서

요구사항에 대하여 엑터와 시스템 활동 및 상호간의 활동에 대하여 순서화된 시나리오를 기술하고 연관관계를 도식화한 다이어그램
```

```
프로젝트 계획단계에는

WBS를 이용해야함
(Work BreakDown Structure)
프로젝트 일정을 관리하기 위해 단계별 활동내역을 정의한 후 활동간의 선후행관계, 수행기간을 정의하여 일정계획을 수립하고 활동별 담당자를 배정하여 만든 공정계획 문서
```

```
프로젝트 일정계획 수립 시 고려해야 될 사항은 ...

개발 내역에 대한 기술적인 난이도를 사전에 평가해야 하고..
업무를 담당하는 팀원 각각의 능력을 고려해야함..
개발 과정에서 예상치 못한 이슈가 발생하는 경우가 많기 때문에...
이슈를 해결할 버퍼를 고려해서 일정을 계획하는 것이 바람직함
(Critical Chain Method)
```

```
설계가 매우 중요함

기능, 데이터, UI, 아키텍처 요구사항 등을 고려한 체계적인 설계 !
불필요한 코드를 줄이고 테스트와 재작업이 쉬운 코드를 만듬 !
개발과정에서 무한 수정/ 변경 등을 방지!
향후 수정 및 유지보수가 용이하여 운영 생산성이 향상 !
부실한 설계는 프로젝트에서 고정 지연과품질 저하를 야기시키는 요인 !!
```
