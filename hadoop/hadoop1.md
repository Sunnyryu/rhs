분산 컴퓨팅 필요 조건


1. 장애 허용 -  분산 클러스터 노드 중 하나가 문제가 생겨도 메인 컴퓨팅 프로세스에  부정적인 영향 주지 않아야 한다. (프로세스 실패가 발생하지 않아야 한다)
2.복구능력 - 분산 클러스터 노드에서 수행중인 작업이 실패하더라도 작업으로부터 어떤 데이터도 손실 되어서는 안된다.
3.선형적 확장성 - 컴퓨팅 능력, 스토리지 공간 확장등, 성능도 선형적으로 증가해야 합니다.
```

```
하둡 아키텍쳐 : HDFS, Yarn, MapReduce, API
하둡 클러스터 : 하둡분산파일시스템(HDFS)과 클러스터 되어있는 노드 (클러스터 리소스 매니저 - Yarn)를 기반으로 하는 하둡 소프트웨어를 사용하는 컴퓨터들의 집합

크기와 상관없이 구조는 동일하다. !!

마스터 노드 (하둡 2.0부터 마스터 노드 2개 이상 구현 가능하여 고가용성(HA)을 지원)

마스터 노드 (Active, Standby / zookeeper도 함꼐 구성)
@ 하둡 클러스터의 작업을 중재함
@ 하둡 클라이언트는 파일을 저장, 일고, 처리하려면 마스터 노드에 접속해야함
@ namenode가 구성되고, 파일이름 저장, 쓰기 요청에 대해서 파일 시스템의 메타 정복 관리
@ mapreduce 작업의 중재하는 프로세스 JobTracker가 구성


워커노드(슬레이드 노드) : 마스터 노드의 지시를 받아 명령을 수행
실제 data를 저장하고 데이터를 처리 프로세싱하는 노드, TaskTracker)
```

```
hdfs는 hdfs의 스토리지를 관리
Namenode : hdfs 상의 파일 시스템 디렉토리 트리와 파일의 위치 등 hdfs 스토리지 관련 메타 정보를 관리(블럭 데이터를 데이터 노드에 메핑된 것)
파일 디렉토리, 생성, 열기, 쓰기 오퍼레이션 수행
어떤 데이터 노드에 복제되고, 복제 후에 삭제할지 결정
데이터 노드에서 보내온 하트비트와 블럭 리포트를 처리 (block 위치 유지, datanode 상태 관리)

SecondaryNameNode : (한시간 간격으로) - hdfs 스토리지 메터 정보 업데이터 (기본 1시간 간격, fsimage 파일과 editlog파일을 merge)


DataNode - 마스터 노드에 접속 유지, 3초 간격으로 heartbeat, block report를 주기적으로 전송, 마스터 노드의 요청을 처리(block 저장, block 삭제)
로컬 파일 시스템에 블록을 저장, 데이터 에 대한 읽기 , 쓰기 수행
데이터 블록 생성 및 삭제 수행 , 클러스터에 데이터 블럭 복제, 주기적으로 하트비트와 블럭 리포트 전송
yarn 서비스
resource manager / node manager / application manager

resource manager : 마스터 노드에서 실행, 클러스터의 리소스를 나눠주는 역할, TaskTracker들의 Taske들에 대한 스케쥴링함

node manager : work node에서 실행, task들을 실행시키고 관리, resouce manager와 관계 유지, test 상태, 노드 상태 관리
application manager - 실제 cluster 상에서 메인이 되는 마스터 프로세스로서 어플리케이션 별로 하나씩 실행됨, 클러스터에서 실행되는 어플리케이션의 실행 조율, 리소스 매니저와 통신(관계 유지)하면서 리소스 조절
```

```
하둡 클러스터에서 장애 허용과 복구 능력을 위해 sharding / (분산), replication을 수행한다.

배치 처리, 파일 기반 처리 (map의 처리 결과도 map처리된 datanode에 파일로 저장, reducer의 출력결과도 hdfs에 저장, disk 기반, stream 기반, sequential하게 처리됩니다. )
